# v0.1.0 RAG 检修问答最小闭环

## 版本主题

本版本完成设备检修知识库的第一条可演示主链路：资料入库、文本解析、chunk 切分、真实 embedding 向量化、语义检索和 RAG 检修问答。

## 已完成能力

- 支持 PDF、TXT、Markdown 文档上传，并记录文档状态。
- 支持文档解析为纯文本，保存到本地解析目录。
- 支持固定长度 chunk 切分，并将 chunk 写入 SQLite。
- 支持真实 embedding 接入，当前已验证 `siliconflow` + `BAAI/bge-m3`。
- 支持本地持久化向量检索，返回 chunk 内容、分数和来源 metadata。
- 支持 `POST /api/chat/repair` RAG 检修问答接口。
- 支持 OpenAI-compatible LLM 调用，当前已验证 `deepseek` provider。
- RAG 输出包含故障理解、可能原因、排查步骤、安全注意事项和系统追溯 sources。
- 新增 `scripts/test_retrieval.py` 与 `scripts/test_rag_chat.py`，用于快速生成检索和问答测试报告。

## 重要技术改进

- `embedding_service.py` 统一封装 mock 与真实 embedding，业务代码不依赖具体供应商。
- `llm_service.py` 统一封装 OpenAI-compatible chat completion，支持后续切换模型。
- `rag_service.py` 固定 sources 来源为检索结果 metadata，避免由模型编造来源。
- `.env.example` 增加 embedding 与 LLM 配置项，`.env` 保持本地私密配置。
- `.gitignore` 隔离本地依赖、缓存、SQLite、向量库和运行时上传副本。

## 已知限制

- 当前 chunk 策略仍是固定长度加 overlap，可能把多个章节或故障案例混入同一 chunk。
- 当前向量库为轻量本地 JSON 持久化实现，保留 ChromaStore 边界，后续可替换为真正 Chroma。
- 前端目前重点覆盖知识库入库链路，RAG 问答主要通过脚本和接口验证。
- 检修记录、复杂权限、多 Agent 和前端美化暂未纳入本阶段。

## 验收路径

1. 准备 `.env` 中的 embedding 和 LLM 配置。
2. 运行 `scripts/test_retrieval.py`，确认 Top-K 检索结果命中电源、过温、焦味、E07 等相关资料。
3. 运行 `scripts/test_rag_chat.py`，确认问答结果结构化生成，并且 sources 包含 `chunk_id`、`document_id`、`filename` 和 `score`。
4. 通过前端知识库页面验证上传、解析、切分和入库状态流转。

## 版本结论

v0.1.0 已经从“文档处理系统”推进到“可追溯的 RAG 检修问答系统”。下一阶段建议优先补前端问答入口和真实业务资料，再根据检索质量回头优化 chunk 策略。
