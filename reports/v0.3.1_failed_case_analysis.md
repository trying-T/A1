## 重点问题回答

### smoke-001 是否属于 Level 0/1 边界标签问题？

是。该题问火花塞“检查和安装”，不是高风险安全边界，也不需要工单。当前系统判 Level 1，说明它把“检查/安装”视为轻量操作相关；这比 expected Level 0 更保守，但不构成业务错误。建议作为 `eval_label_issue`，把 expected 调整为 Level 1 或允许 Level 0/1。

### insufficient-* 是否需要单独引入 basis_status？

需要。当前报告显示资料不足题全部能提示不足：`6/6`，但 risk_level 表现不稳定：有的 Level 0，有的 Level 1，有的 Level 2。这说明“资料是否足够”不应由 risk_level 承担。建议新增：

- `basis_status`: `sufficient / partial / insufficient`
- `basis_reasons`: 说明缺少具体型号、第三方材料兼容性、未知故障码、缺少端子图等
- 当 `basis_status=insufficient` 时，WorkOrder 可 recommend，但不应直接 ready

### insufficient-003 为什么 ready_to_create=True？

因为当前 WorkOrder quality gate 主要检查 payload 是否有 fault_symptom、source_chunk_ids、inspection/repair_steps、safety_actions 等结构字段。该题虽然文本已检测到资料不足，但这个“依据不足”没有成为 ready 阻断条件。因此系统可以同时出现 `detected_insufficient_basis=True` 和 `ready_to_create=True`。

建议后续把 `basis_status=insufficient` 纳入 WorkOrder quality gate，阻断 ready，并在 missing_fields 中加入 `sufficient_manual_basis`。

### docintent-005 为什么 recommend/ready=True？

题目包含“AH3、上控制柜风扇、更换与维护”等强维修词，系统把它当成实际维修任务，而不是“这个内容位于哪类维护内容中”的文档定位题。也就是说，规则看到了维护对象和章节名，但没有识别“位于哪类内容中”这种资料查询意图。

建议后续为“文档定位/章节归属/手册包含哪些内容/位于哪章”增加降权，覆盖维护词但不触发工单 ready。

### risk-syn-005 和 docintent-001 为什么 FANUC 场景被 bfp 文档压过？

两题都与 FANUC 安全相关，但 Top-3 被 bfp 占据，说明当前 rerank 对 FANUC 的识别或 boost 不足以抵消 bfp 中文安全手册的相似度优势。

- `risk-syn-005` 使用中文“防护门、旁路、围栏、示教定位”，没有直接出现 `safety fence`，容易被 bfp 的中文安全栅栏/互锁内容吸走。
- `docintent-001` 虽然有 `FANUC` 和 `emergency stop`，但 bfp 的安全控制资料仍排在 Top-3，说明强品牌实体需要更高权重，或 rerank debug 中 preferred document 未正确压过相似中文资料。

建议后续只优化 DocumentIntent/rerank：补 FANUC 中文同义词，提高强品牌实体 boost，仍保持不强制过滤其他 sources。

## 建议处理顺序

1. 先调整明显 eval label：`smoke-001`、`wo-boundary-002`。
2. 引入 `basis_status` 设计稿，不急着改 SafetyGuard。
3. 再治理 WorkOrder ready：资料不足时 recommend 可以为 true，但 ready 应 false。
4. 最后优化 DocumentIntent：补 FANUC 中文同义实体和强品牌 boost，重点回归 `risk-syn-005`、`docintent-001`。
